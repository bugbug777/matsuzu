什麼是前端？如果只是單看這樣的一個文字段落，你可能會不知道這是什麼意思，但是既然你點進來了，也許你是搜索了「前端」之類的關鍵字，這才把你帶到這個地方，而且你可能是對於建立網站有點興趣，沒錯這裡確實是要討論關於什麼是網站前端，但我之所以這樣陳述，那是因為許多東西都有類似的概念。

<!--more-->

舉例來說，在一般餐飲環境，都會區分前場、後場，前場可能是服務客人的端盤小哥，或者負責收銀的櫃檯女店員，後場可能是洗菜、洗碗盤的打雜小弟，也可能是料理食材的廚師。

之所以會這樣區分，是因為任務性質的不同，負責料理的廚師，可能不是那麼會與客人交際，也就比較不擅於與人構通（當然這並不是一定如此），而端盤子的小哥，可別因為端盤子這樣簡單的任務就小看他，這個小哥可能很會逗客人笑，客人一開心就多點了幾道菜，換句話說，小哥很擅於與人溝通，而且餐廳的前場通常會安排面容比較姣好的服務生（當然這也並不是一定如此），我想這很容易理解。

當然，也不一定每間餐廳都是這樣，也有可能老闆很厲害，自己一個人前、後都包了，這也不是不無可能。

你可能會好奇，明明是來學怎麼建立網站的，突然聊起餐廳來了，其實很多東西的概念都是很雷同的
，這裡只是簡單的舉個例子而已，這會對你建立什麼是一個「網站」的概念，非常有幫助。

話說回來，倒底什麼是前端？前端其實就是你現在透過瀏覽器看到的這個頁面，我們一般也稱為「網頁」，看起來就像書中的內容一樣，一頁一頁的，而不同的連結可以帶你到不同的頁面。

而一個「網站」通常會有使用者可以觀看內容的前端，以及將這些內容整理、搜集、處理的後端，在比較早期的網路世界，其實沒有所謂的前、後端的區分，由於建置網站的過程並沒那麼高度的專業分工，而我們也給這樣的一個狀態，也就是前、後端全包的形式一個稱呼，全端。

看到這邊你會驚訝地發現，我們的網站跟一家餐廳的結構何其相似，如果你的餐廳只有好的服務，好看的店員，但是餐點料理非常差勁，我想你肯定不會想再次光顧，但是反過來說，如果今天一家餐廳東西真的很好吃，但是服務的態度非常差勁，店員 ⋯⋯，嗯，這樣說不太好，可能沒吸引上門光顧的一個契機，當然這個比喻不是太貼切，但是我想你明白我想表達的是什麼，太過極端的案例都不是太好的例子，但是我們當前會比較著重在前端的部分，也就是怎麼將一個好的網站呈現給我們的客人看。

## 網站是怎麼運作的？

一般人可能會使用瀏覽器，透過搜尋引擎或者其他的方式，點擊連結，然後連到你的網站，瀏覽裡面的內容，但是你可有曾經思考過，這樣一個看似稀鬆平常的過程，幾乎每天都會重複進行的事情，尤其是每天一定滑個幾小時手機的你，但其實這樣的一個過程一點也不困難。  

![request v.s. response](/images/tutorial/frontend/request_response.png)

HTTP 是一種網路協定，在網路世界傳遞資料，就如同在現實世界傳遞資料，我們需要知道對方說的是什麼話，是說英文、中文還是日文，通過這樣的協定，我們才有辦法在網路世界彼此傳遞訊息，並且進一步的解讀訊息後進行處理，而 HTTP 僅僅只是眾多協定之一。

通過 HTTP 向伺服器發送一個請求（ request ），就像是在對伺服器說：「嗨嗨，安安給虧嗎？幾歲？住哪？」（大誤），不不不，不是這樣的，我們這麼紳士是不會這麼粗俗的，發送這樣的一個請求，更像是跟某某網路店家下訂單：「你好，某某店，我想要跟妳買什麼什麼」，店家接受你的請求後，接著依照你的請求，以及他們店裡的一些規定，然後將你要的貨物一起寄回去給你，回傳一個響應（ response ），接著你就可以在瀏覽器上看到想要的網頁內容。

一般來說，我們會稱發送請求的這一端為客戶端（ client ），而接收請求並回傳響應的這一端，通常也會是伺服器，稱其為伺服端（ server ）。

但是這一個架構可以簡單的延伸一下，request 與 response 是一個相對的架構。今天我們在後端，也就是伺服器這一端，也是有可能發送其他請求，去請求其他服務，例如：金流服務，這個時候的伺服器端，也可以是一個客戶端。同時，被請求的金流服務的伺服器，就會是一個伺服器端。

```txt
client    server
瀏覽器 <=> 伺服器
          伺服器 <=> 金流伺服器
          client    server
```

大概是這樣的一個感覺！
